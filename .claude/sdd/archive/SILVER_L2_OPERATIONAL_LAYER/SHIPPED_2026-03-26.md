## Summary

Introduced a formal Silver L2 layer in the PSP payment system's Medallion Architecture, creating 4 domain-scoped DLT materialized views (settlement ops, customer service, risk operations, merchant operations) that eliminate the need for data analysts to write complex multi-join queries against Silver L1 tables. Migrated all 3 Gold aggregated views to read from domain-specific L2 tables instead of a monolithic 200+ column unified table, and deprecated the legacy `silver_unified_transactions` table. End-to-end validated with ShadowTraffic data generation, DLT pipeline execution (22 notebooks), and FlowCheck CLI validation.

## Lessons Learned

### Process

- The Brainstorm phase (6 discovery questions) prevented scope creep — YAGNI removed 5 features that would have added complexity without value
- Pre-validating requirements in Brainstorm gave Define a perfect 15/15 Clarity Score with zero additional clarification rounds needed
- Single-session delivery (brainstorm through ship) was possible because the scope was well-contained — 4 new SQL files + 5 modifications

### Technical

- **DLT Materialized Views can reference Streaming Tables by catalog name** — this avoids the Python complexity required for multi-way streaming JOINs. The existing `unified_transactions.py` used Python because `STREAM()` doesn't support mixed join types, but MVs don't need `STREAM()`
- **Column name mismatches are the #1 migration risk** — the Gold `risk_fraud_monitoring` view referenced `psp_revenue` (a derived alias from the unified table), which didn't exist in the L2 table. The L2 had `fees_total_amount` instead. This was caught immediately by the DLT pipeline and fixed in one edit
- **Customer_service L2 needed a merchant JOIN** not anticipated in the DESIGN — the Gold `customer_analytics` aggregates `COUNT(DISTINCT merchant_id)` and `COUNT(DISTINCT merchant_country)`, requiring merchant context in the L2 source. Always trace Gold column references back to their source before migration
- **Aggregated grain L2 tables (merchant_operations)** significantly simplify Gold views — the Gold `merchant_performance` CTE was completely eliminated because L2 already pre-aggregated at the same grain

### Communication

- One question at a time during Brainstorm kept the dialogue focused — all 6 answers were clear and decisive
- The problem statement was refined three times (from vague "copy data" to specific "monolithic L2 too wide") through the question flow

### Tools & Libraries

- **ShadowTraffic** with Azure Blob Storage output worked well for synthetic data generation — 7 entity streams with referential integrity
- **Databricks Asset Bundles (DABs)** made deploy+run seamless — `bundle deploy` + `bundle run` with `--target dev` parameter
- **FlowCheck CLI** validated the full pipeline health across all layers in seconds
- **DLT dependency resolution** automatically ordered L2 before Gold without any explicit configuration

## Recommendations for Future Work

| Area | Recommendation |
|------|----------------|
| Retire unified table | After confirming all consumers have migrated, remove `unified_transactions.py` from the pipeline to save ~6 minutes of pipeline runtime |
| Add missing Gold columns | Gold `merchant_performance` lost `daily_subtotal`, `daily_tax`, `daily_tips`, `orders_with_tips`, `high_value_orders`, `avg_tip_rate`, `avg_tax_rate` during migration — add to L2 if needed |
| L2 data quality monitoring | Add DLT expectations directly on L2 views for join-key validation (currently relies on L1 expectations) |
| FlowCheck L2 support | Extend FlowCheck CLI to trace Silver L2 tables — currently it traces Bronze → Silver L1 → Gold, skipping L2 |
| Additional L2 domains | If new operational domains emerge (e.g., compliance ops, billing ops), follow the same pattern: SQL MV in `silver/l2_{domain}.sql` |
